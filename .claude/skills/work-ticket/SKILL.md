# Work a ticket

1. **Pick.** Given an id, run `interfacile show <ID>`. Given nothing, run
   `interfacile ready` and propose the top item — it is sorted by priority.
   Never start a ticket that `interfacile deps <ID>` shows waiting on an open
   dependency.
2. **Read.** The acceptance criteria are the contract. If they are vague, or
   the approach no longer makes sense, update the ticket body first and confirm
   the change with the user before writing code.
3. **Implement.** Keep changes scoped to the ticket. When you find unrelated
   problems along the way, file them with the `new-ticket` skill instead of
   fixing them here.
4. **Verify.** Check off each `- [ ]` criterion in the ticket body only after
   you have actually verified it (tests pass, the feature demonstrably works).
5. **Close.** When every criterion is checked, follow the `close-ticket`
   skill. Reference the ticket id in the commit message, e.g.
   `IF-0012: short summary of what shipped`.
